The Big Picture

What does this platform do?

Think of ObsLens as a window into your running applications. Instead of adding console.log everywhere, your apps send structured data about what they're doing — and ObsLens organizes, stores, and visualizes all of it.

📋

Logs

Every message your application writes — errors, warnings, info — gets collected, searchable, and filterable. Find that one error needle in a haystack of millions of log lines.

🔗

Distributed Traces

When a user clicks a button, the request might touch 5 different services. Traces stitch that entire journey together into a visual timeline, so you can see exactly where time is spent.

📈

Metrics

Numbers that change over time — request counts, response times, error rates, CPU usage. Metrics tell you the health of your system at a glance through charts and graphs.

🔔

Alerts

Set rules like "tell me if error rate goes above 5%" and ObsLens will watch your data and notify you when something goes wrong — before your users notice.

📊

Dashboards

Build custom views that combine multiple charts and tables into one screen. Create a dashboard for each team or service to monitor what matters most to them.

📦

Simple SDK

Adding observability to your app takes just two lines of code. The SDK automatically captures most data for you — no manual work needed for common frameworks.

Workflow

The data journey, step by step

From the moment your application does something interesting to the moment you see it on your screen — here's every step along the way.

💻

1. Your App Generates Telemetry

When your application handles a request, the SDK automatically captures what happened: how long it took, which services were called, any errors that occurred, and log messages you write. This happens in the background with near-zero overhead.

@obs/sdk · OpenTelemetry
📡

2. Data Arrives at the Collector

Your app sends data over the network to the OpenTelemetry Collector — a central hub that receives telemetry from all your services. The collector groups small messages into larger batches (more efficient) and adds useful metadata.

OpenTelemetry Collector · OTLP protocol
📦

3. Messages are Queued for Safety

The collector places logs and traces onto a message queue (Redpanda). Think of it as a conveyor belt — even if the database is busy for a moment, no data gets lost. Metrics take a shortcut and go directly to VictoriaMetrics for real-time availability.

Redpanda (Kafka-compatible)
🗃

4. Consumer Writes to Storage

A consumer service reads from the message queue and efficiently batch-inserts data into the databases. Logs and traces go to ClickHouse (a high-compression columnar database perfect for searching). Metrics are already in VictoriaMetrics from step 3.

ClickHouse · VictoriaMetrics
🔍

5. Query Service Makes Data Accessible

The Query Service provides a clean REST API that the dashboard calls. It translates your searches ("show me all errors from the payment service in the last hour") into optimized database queries and returns the results.

Fastify · REST API · PromQL
🔔

6. Alert Service Watches for Problems

Every 30 seconds, the Alert Service checks your rules against the data. "Is the error rate above 5%?" "Are there more than 100 failed logins?" When a condition is met, it fires a notification so you know immediately.

Rule evaluator · Webhook notifications
🎨

7. You Explore on the Dashboard

The React dashboard ties everything together. Browse logs, click into a trace to see its waterfall timeline, chart metrics with PromQL, manage your alert rules, and build custom dashboards with drag-and-drop widgets.

React · Tailwind CSS · Recharts
Developer Experience

Add observability in two lines

The SDK wraps OpenTelemetry into a simple, batteries-included package. Initialize once and your app automatically captures HTTP requests, database calls, and more.

// Initialize once at the top of your app import { init, logger, metrics, tracer } from "@obs/sdk"; init({ serviceName: "my-service" }); // That's it! Now use the helpers anywhere: logger.info("User signed up", { userId: "123" }); const counter = metrics.counter("signups.total"); counter.add(1); await tracer.withSpan("send-welcome-email", async (span) => { // ... your code here — span auto-records duration & errors });

📋 logger

Structured logging that automatically links to the current trace. Find the exact log for a specific request.

logger.error("Payment failed", { orderId: "456", reason: "timeout" });

📈 metrics

Create counters, gauges, and histograms to track numbers over time. Query them later with PromQL.

const latency = metrics.histogram( "request.duration_ms" ); latency.record(42.5, { path: "/api" });

🔗 tracer

Create custom spans for business logic. Most HTTP/DB tracing is automatic — use this for important operations.

await tracer.withSpan("checkout", async (span) => { span.setAttributes({ cartSize: 3 }); // auto-captures errors & duration } );

Decision Why We Chose It
Redpanda over Kafka Same API, but runs as a single binary with much lower memory usage — perfect for running on a laptop without burning through resources.
ClickHouse for logs & traces Columnar storage compresses log data to 1/10th the size of Elasticsearch. Searching through millions of logs is fast because it only reads the columns you ask about.
VictoriaMetrics for metrics Speaks the same language as Prometheus (PromQL), but compresses data to ~1 byte per data point and uses very little RAM. Drop-in compatible.
OTel Collector as the hub Industry standard, vendor-neutral. Your apps talk one protocol (OTLP), and the collector handles routing to different backends. Change storage without changing app code.
Fastify for backend services 2x faster than Express with built-in JSON schema validation. TypeScript-first, lightweight, and battle-tested in production.
pnpm + Turborepo Monorepo that installs fast and runs tasks in parallel. Shared types between frontend and backend catch bugs at compile time, not in production.
Get Started

Running locally in 5 steps

Everything runs on your machine. No cloud accounts, no API keys, no credit cards. Just Docker and Node.js.

1

Install dependencies

Clone the repository and install all packages across the monorepo.

pnpm install
2

Start infrastructure

This boots up Redpanda, ClickHouse, VictoriaMetrics, and the OTel Collector in Docker containers.

make infra
3

Start all services

Launches the ingest service, query service, alert service, and the React dashboard — all with hot reload.

make dev
4

Run the demo app & generate traffic

The demo app simulates a real service sending telemetry. The load generator creates realistic request patterns with successes, errors, and slow responses.

make demo

pnpm --filter @obs/demo-app load
5

Open the dashboard

Visit your browser and start exploring your data — logs, traces, metrics, alerts, and dashboards are all there.

http://localhost:3000

🐋 Prefer Docker only? No local Node.js needed.

Run the entire platform — all services and infrastructure — in containers with a single command:

docker compose up --build

🛠 Other useful commands

Type-check everything make typecheck
Run tests make test
Stop infrastructure make infra-down
Full reset (erase data) make reset
